# 🚂 راهنمای Deploy کامل LeoBot روی Railway

این راهنما مخصوص Deploy کردن LeoBot از روی ریپوی گیت‌هاب روی Railway است
و باگ‌های شناخته‌شده‌ی «بات پاسخ نمی‌دهد» و «پنل ادمین با دکمه کار
نمی‌کند» را هم پوشش می‌دهد (رفع این باگ‌ها در کد همین نسخه انجام شده).

---

## ۰) قبل از هر کاری: امنیت توکن

فایل `.env` در حال حاضر **داخل ریپوی گیت‌هاب کامیت شده** و ریپو Public
است. یعنی هر کسی که ریپو را دیده (یا هر ربات خودکاری که گیت‌هاب را
برای اسکن اسرار می‌گردد) به توکن بات و رمز پنل شما دسترسی داشته.

**قبل از ادامه، این کارها را انجام بده:**

1. از داخل اپ روبیکا، بخش مدیریت بات‌ها، **توکن فعلی را Revoke/بازتولید کن**
   و یک توکن جدید بگیر (حتی اگر همین الان یک توکن جدید به من دادی، همان
   را هم اگر جای دیگری (چت، اسکرین‌شات و ...) به اشتراک گذاشته‌ای، از نو
   بازتولید کن تا مطمئن باشی فقط در Railway Variables زنده است).
2. `.env` را از ریپو حذف کن (نه فقط پاک کردن محتوا، بلکه از Git هم):
   ```bash
   git rm --cached .env
   git commit -m "remove .env from version control"
   git push
   ```
3. فایل `.gitignore` جدید (همراه این تغییرات اضافه شده) را کامیت و
   پوش کن تا دیگر هیچ‌وقت `.env` کامیت نشود.
4. توکن و رمزها را **فقط** در بخش Variables پروژه‌ی Railway وارد کن،
   نه در کد یا فایل کامیت‌شده.

---

## ۱) ساخت پروژه در Railway

1. وارد [railway.app](https://railway.app) شو و با گیت‌هاب لاگین کن.
2. **New Project → Deploy from GitHub repo** را بزن و ریپوی
   `tcmmc/LeoBot` را انتخاب کن.
3. Railway به‌طور خودکار `railway.json` را می‌بیند و از Nixpacks
   برای build استفاده می‌کند (فایل `requirements.txt` را تشخیص
   می‌دهد و پایتون را طبق `.python-version` نصب می‌کند).
   - اگر ترجیح می‌دهی با Docker بیلد شود: در تنظیمات سرویس،
     Settings → Build → Builder را روی **Dockerfile** بگذار
     (فایل `Dockerfile` همراه این تغییرات اضافه شده).

---

## ۲) تنظیم Variables

در سرویس ساخته‌شده، برو به تب **Variables** و مقادیر زیر را اضافه کن
(روی «Raw Editor» بزن تا همه را یک‌جا پیست کنی):

```
BOT_TOKEN=<توکن جدیدی که در مرحله‌ی ۰ گرفتی>
OWNER_ID=u0EGSTT0a21b558e610d3e700ce8f2fa
PANEL_ADMIN_IDS=
CREATOR_IDS=
ADMIN_PASSWORD=GG777mahi
DATABASE_FILE=leobot.db
LOG_FILE=leobot.log
LOG_LEVEL=INFO
DEFAULT_WARN_LIMIT=3
DEFAULT_FLOOD_LIMIT=6
DEFAULT_FLOOD_WINDOW=8
DEFAULT_DUPLICATE_LIMIT=3
PANEL_MAX_LOGIN_ATTEMPTS=3
PANEL_LOCKOUT_SECONDS=300
RECONNECT_DELAY_SECONDS=5
RECONNECT_MAX_DELAY_SECONDS=60
```

نکته‌ی مهم درباره‌ی `OWNER_ID`: مقداری که فرستادی
(`u0EGSTT0a21b558e610d3e700ce8f2fa`) با پیشوند `u0` شروع می‌شود که
دقیقاً قرارداد GUID کاربر در روبیکاست — این خودش یک نشونه‌ی خوبه که
شناسه‌ی درستی دادی (برخلاف GUID گروه که با `g0` شروع می‌شود).

**برای PYTHONUNBUFFERED** (تضمین دیده‌شدن فوری لاگ‌ها در Railway):
```
PYTHONUNBUFFERED=1
```
این را هم به Variables اضافه کن؛ کد هم خودش تلاش می‌کند stdout را
Unbuffered کند، ولی این متغیر تضمین سطح پایین‌تری می‌دهد.

## ۳) (اختیاری ولی توصیه‌شده) اضافه کردن Volume برای ماندگاری دیتابیس

فایل‌سیستم پیش‌فرض Railway **Ephemeral** است — یعنی با هر Deploy یا
Restart، فایل `data/leobot.db` (شامل تنظیمات گروه‌ها، اخطارها، لیست
کاربران و ...) از بین می‌رود.

1. در سرویس، تب **Volumes → New Volume** را بزن.
2. Mount Path را روی `/data` بگذار.
3. در Variables، مقدار زیر را اضافه کن:
   ```
   DATA_DIR_PATH=/data
   ```
4. Redeploy کن. از این به بعد دیتابیس و لاگ‌ها روی Volume ذخیره
   می‌شوند و با هر Deploy پاک نمی‌شوند.

---

## ۴) Deploy و بررسی لاگ‌ها

بعد از تنظیم Variables، Railway به‌طور خودکار Deploy را شروع می‌کند.
برو به تب **Deployments → View Logs** و دنبال این خط‌ها بگرد (همه‌شان
باید ظاهر شوند؛ اگر یکی نبود، دقیقاً همان‌جا مشکل است):

```
[INFO] [database] دیتابیس با موفقیت مقداردهی اولیه شد.
[INFO] [main] BOT_TOKEN بارگذاری شد (...) — طول: NN کاراکتر
[INFO] [client] هندلر دریافت پیام ثبت شد (حالت: ...)
[INFO] [client] هندلر دریافت پیام با موفقیت روی BotClient ثبت شد.
[INFO] [events] Event handler های LeoBot با موفقیت ثبت شدند.
[INFO] [main] LeoBot با موفقیت روشن شد و در حال گوش‌دادن به پیام‌هاست.
```

اگر خط «هندلر دریافت پیام ثبت شد (حالت: ...)» را دیدی، به مقدار
داخل پرانتز نگاه کن:
- `no-filter` → بهترین حالت، همه‌چیز باید کار کند.
- نام فیلترهای دیگر (`private`, `group`, ...) → یعنی نسخه‌ی rubpy
  نصب‌شده فقط با فیلتر مشخص کار می‌کند؛ کد به‌صورت خودکار روی همه‌ی
  فیلترهای پیداشده جداگانه ثبت می‌کند، پس باید هم PV هم گروه کار کند.

اگر هیچ‌کدام از این خط‌ها دیده نشد یا خطای `RuntimeError: امکان ثبت
هیچ هندلری ...` دیدی، یعنی نسخه‌ی rubpy روی Railway با کد ناسازگار
است؛ `requirements.txt` مقدار `rubpy>=7.1.24` را Pin کرده، اما اگر
باز هم مشکل بود، در تب Deployments → Redeploy را با پاک کردن Cache
بزن تا آخرین نسخه نصب شود.

## ۵) تست در PV و گروه

### تست PV
1. با همان اکانتی که `OWNER_ID` را از آن گرفتی، به PV بات برو و پیام
   بفرست (هر پیامی، یا دکمه‌ی Start را بزن).
2. باید پیام خوش‌آمد + دکمه‌ی «🔐 پنل ادمین» را ببینی.
3. رمز `GG777mahi` را بفرست → پنل باید باز شود.

اگر پیام خوش‌آمد را دیدی ولی دکمه را ندیدی، یعنی `OWNER_ID` در
Variables را اشتباه/متفاوت از GUID واقعی‌ات وارد کرده‌ای — در لاگ با
`LOG_LEVEL=DEBUG` دنبال خط `دیسپچ آپدیت -> ... user_guid=...` بگرد و
مقدار واقعی `user_guid` را با `OWNER_ID` مقایسه کن.

### تست در گروه
1. بات را به یک گروه اضافه کن.
2. در گروه بنویس: `تنظیمات` یا `قفل لینک`.
3. اگر پاسخی نیامد، `LOG_LEVEL=DEBUG` را در Variables بگذار،
   Redeploy کن، دوباره پیام بفرست، و در لاگ دنبال خط
   `آپدیت خام دریافت شد: {...}` بگرد — این خط دقیقاً ساختار خامی که
   rubpy فرستاده را نشان می‌دهد و مشخص می‌کند چه فیلدهایی برای
   `chat_guid`/`user_guid` پیدا نشده.

**نکته:** بعد از عیب‌یابی، `LOG_LEVEL` را دوباره روی `INFO` برگردان
تا حجم لاگ زیاد نشود.

## ۶) رفع خطاهای رایج مخصوص Railway

| علامت | علت محتمل | راه‌حل |
|---|---|---|
| Deploy موفق ولی بات هیچ پاسخی نمی‌دهد | نسخه‌ی rubpy امضای هندلر متفاوتی صدا می‌زند | این نسخه‌ی کد این مشکل را با هندلر `*args` و Exception Handler سراسری روی asyncio رفع کرده؛ در لاگ دنبال «خطای بی‌صدای asyncio شناسایی شد» بگرد |
| دکمه‌ی «پنل ادمین» کار نمی‌کند | ساختار `aux_data` در نسخه‌ی نصب‌شده متفاوت است | با `LOG_LEVEL=DEBUG`، خط «آپدیت خام دریافت شد» را برای کلیک دکمه پیدا کن و ساختار واقعی را ببین؛ در صورت نیاز `utils/helpers.py::get_button_id` را با نام فیلد واقعی به‌روزرسانی کن |
| لاگ‌ها با تأخیر زیاد یا اصلا در Railway دیده نمی‌شوند | Buffering خروجی پایتون | `PYTHONUNBUFFERED=1` را در Variables اضافه کن (کد هم خودش تلاش برای رفع این مشکل می‌کند) |
| با هر Restart، تنظیمات گروه/اخطارها از بین می‌رود | فایل‌سیستم Ephemeral Railway | طبق بخش ۳ یک Volume وصل کن و `DATA_DIR_PATH=/data` را تنظیم کن |
| Deploy Crash Loop می‌شود | مقدار `BOT_TOKEN` یا `OWNER_ID` خالی/غلط است | خطای `[CONFIG] مقدار الزامی ...` را در لاگ ببین؛ همان متغیر را در Variables چک کن |
| `ModuleNotFoundError` هنگام Start | Build کش قدیمی یا `requirements.txt` به‌روزرسانی نشده | Deployments → سه‌نقطه → Redeploy، یا Settings → Clear Build Cache |
| «بن» عضو را واقعاً اخراج نمی‌کند | محدودیت شناخته‌شده‌ی Bot API رسمی روبیکا (نه باگ) | توضیح کامل در `INSTALL.md` بخش «نکته‌ی فنی مهم» — پیام‌های کاربر خودکار حذف می‌شوند اما اخراج واقعی نیاز به اقدام دستی دارد |

## ۷) خلاصه‌ی فایل‌های جدید/تغییریافته برای Railway

```
LeoBot/
├── Procfile              دستور اجرا برای Railway/Heroku-style بیلدرها
├── railway.json           تنظیمات Build/Deploy/Healthcheck Railway
├── Dockerfile              بیلدر جایگزین (اختیاری)
├── .dockerignore
├── .python-version         پین نسخه‌ی پایتون (3.11) برای Nixpacks
├── .gitignore              جلوگیری از کامیت مجدد .env و فایل‌های محلی
└── utils/healthcheck.py    سرور HTTP سبک برای Healthcheck روی $PORT
```

موفق باشید! اگر بعد از این مراحل هنوز پاسخی نگرفتی، خروجی لاگ Railway
(با `LOG_LEVEL=DEBUG`) را برام بفرست تا دقیقاً ساختار خام پیام‌هایی که
rubpy می‌فرستد را ببینم و `utils/helpers.py` را برای نسخه‌ی دقیق شما
Fine-tune کنم.
